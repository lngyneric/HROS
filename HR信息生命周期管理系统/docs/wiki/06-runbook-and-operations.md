# 06. 运行与运维指南

## 6.1 仓库使用场景

本仓库主要适用于以下场景：

- 作为 HR 信息系统的“数据与文档标准库”
  - 定义组织、职位、员工信息、流程结构、表单模板等。
  - 为后续应用系统、脚本、数据仓库提供统一的字段和结构规范。
- 作为本地数据处理与验证工具
  - 通过脚本对在职/离职员工信息进行清洗、归档和统计。
  - 通过分析数据层级的 JSON 报告评估数据质量和业务状态。

## 6.2 本地运行数据清洗脚本

### 6.2.1 前置条件

- 已安装 Node.js（建议 16+）。
- 已准备好在职/离职员工信息 JSON 文件：
  - 在职：`05_入离职管理/02_在职期间管理/非敏感信息/在职员工信息.json`
  - 离职：`05_入离职管理/03_离职办理阶段/非敏感信息/离职员工信息.json`

### 6.2.2 修改脚本路径配置

脚本默认使用 Windows 绝对路径，在本地需调整为相对路径或本机绝对路径。示例（相对当前脚本目录）：

```js
const fs = require('fs');
const path = require('path');

const OUTPUT_DIR = __dirname;
const ARCHIVE_DIR = path.join(OUTPUT_DIR, "03_离职办理阶段", "非敏感信息");
const ACTIVE_DIR = path.join(OUTPUT_DIR, "02_在职期间管理", "非敏感信息");
const DATA_SOURCE_ACTIVE = ACTIVE_DIR;
const DATA_SOURCE_RESIGNED = ARCHIVE_DIR;
```

修改后保存。

### 6.2.3 执行脚本

在 `05_入离职管理` 目录下执行：

```bash
node data_cleaning_and_archive.js
```

观察控制台输出，确认：

- 在职和离职员工数量读取正常。
- 数据清洗步骤没有致命错误。
- 归档、索引、统计、报告文件均生成成功。

生成/更新的关键文件包括：

- `02_在职期间管理/非敏感信息/在职员工信息_清洗后.json`
- `02_在职期间管理/非敏感信息/在职员工统计信息.json`
- `03_离职办理阶段/非敏感信息/离职员工归档_*.json`
- `03_离职办理阶段/非敏感信息/归档索引.json`
- `05_入离职管理/数据清洗和归档报告.json`

## 6.3 新增或调整数据文件的建议流程

### 6.3.1 新增元数据定义

当需要新增某类元数据（例如新的员工属性、新的职位等级）时：

1. 在 `01_元数据层级` 中新增或更新相应的定义文档。
2. 更新“员工信息数据表结构”或相关表结构方案。
3. 在业务数据 JSON 中逐步引入新字段，并保证与定义文档一致。

### 6.3.2 新增业务流程数据

对于新的业务流程实例（如新的招聘项目、培训项目、入职批次）：

1. 在对应模块目录下创建新的 JSON 或 Markdown 文档。
2. 遵循命名和字段规范（见《07. 术语表与规范约定》）。
3. 若该数据将参与分析或脚本处理，需同步更新相关脚本或分析逻辑。

## 6.4 版本管理与变更记录

当前仓库通过“数据库更新任务完成报告”“数据库更新文档索引”等 Markdown 文档记录数据结构的变更过程。建议持续遵循以下原则：

- 每一次结构或字段变更：
  - 创建或更新一份变更说明文档，描述变更原因、内容和影响。
  - 在索引文档中新增一条记录，指向详细的变更说明。
- 在后续若接入 Git 等版本控制：
  - 将这些变更说明与实际代码/脚本/SQL 变更的提交对应起来。

## 6.5 安全与权限建议（结合系统总览）

虽然当前仓库主要是本地文件和文档，但从系统总览的安全规范出发，仍建议：

- 对“敏感信息”目录设置更严格的访问控制（例如在共享盘上仅授权必要人员）。
- 对备份文件进行妥善管理，避免敏感数据在备份中泄露。
- 在将来迁移到数据库与应用系统时，严格落实：
  - 角色与权限模型（RBAC）。
  - 传输加密（HTTPS/TLS）和存储加密。
  - 审计日志与异常告警机制。

## 6.6 运维视角的监控点

如果未来将本仓库中的逻辑迁移到服务化环境，建议关注以下监控点：

- 数据清洗与归档任务：
  - 每次任务执行成功/失败情况。
  - 数据质量指标（无效记录数量、缺少字段统计等）。
- 分析任务：
  - 分析结果更新频率（按日/周/月）。
  - 报告生成失败及其原因。
- 存储与权限：
  - 关键目录/数据库空间使用情况。
  - 敏感数据的访问频次与异常访问行为。

当前阶段，这些监控可以通过简单的脚本运行记录和人工检查 JSON 报告来完成，为将来的正式系统运维打基础。

## 6.7 三个世界视角

### 6.7.1 原子世界（有空间 + 有时间）

- 运维动作本身发生在现实世界：谁来执行脚本、何时执行、执行前后如何审批与复核、如何应对突发数据问题。
- 安全与合规要求也属于原子世界约束：人员权限、物理存储介质、共享盘访问规则会决定信息资产能否安全流转。

### 6.7.2 信息世界（无空间 + 有时间）

- 本章的“运行”本质是对信息世界资产的维护：保证数据清洗、归档、报告生成可重复且可追溯。
- 时间性体现为运维的节律与记录：执行频率、变更记录、失败原因、回滚与恢复策略都需要被结构化沉淀。

### 6.7.3 AI 世界（无空间 + 存储层无时间）

- AI 能力落地会放大运维要求：数据漂移、模型版本、特征口径变化都需要被纳入监控与审计。
- 先把信息世界的运行机制做稳（数据链路、权限、变更记录），才能让 AI 世界的推理与预测具备可解释、可复现和可治理的基础。
