# 01. 系统架构说明

## 1.1 逻辑架构总览

从逻辑层面，“HR信息生命周期管理系统”主要分为三大层级：

1. 元数据层级（01_元数据层级）
   - 存放组织架构、职位描述、流程定义、表单模板、员工信息表结构等“配置型数据”。
   - 再细分为“非敏感信息”和“敏感信息”两大块，分别对应一般业务配置与需要严格保护的个人/薪资/绩效/健康数据。
2. 业务流程层级（02~05）
   - 招聘流程管理（02）、培训与发展管理（03）、薪资管理（04，规划中）、入离职管理（05）。
   - 每个模块按业务阶段再细分目录，如“职位发布阶段/简历筛选阶段/面试评估阶段”等。
3. 分析数据层级（06_分析数据层级）
   - 存放各类分析结果和综合报告，例如“员工状态分析报告”“新员工入职分析报告”等 JSON 文件。

目录与层级关系在系统总览文档中已有清晰展示，可参考：

- [HR信息生命周期管理系统总览.md - 2.1 整体架构](file:///Users/cherrych/Documents/trae_projects/HROS/HR信息生命周期管理系统/HR信息生命周期管理系统总览.md#L24-L66)

这三大层级的关系可以理解为：

- 元数据层级：定义“系统中有哪些实体/字段/流程/表单”，类似配置中心。
- 业务流程层级：承载日常 HR 操作产生的“业务数据”，例如员工在职/离职信息、招聘记录、培训记录等。
- 分析数据层级：以元数据为维度、以业务流程数据为事实，生成各类统计表和分析报告。

## 1.3 技术架构：规划 vs 当前实现

系统总览文档中给出了理想状态下的技术架构（参考 [总览 5.1~5.3](file:///Users/cherrych/Documents/trae_projects/HROS/HR信息生命周期管理系统/HR信息生命周期管理系统总览.md#L196-L215)）：

- 前端：React + Ant Design
- 后端：Node.js + Express
- 数据库：PostgreSQL
- 部署：Docker 容器化
- 安全：
  - 认证授权：JWT Token + RBAC
  - 网络安全：HTTPS + 防火墙
  - 数据安全：加密存储 + 访问控制
  - 审计安全：日志 + 监控告警

但在当前仓库中：

- 未发现前端代码（无 React/Ant Design 相关工程）。
- 未发现后端服务代码（无 Express 应用、无 API 定义）。
- 未发现数据库建表脚本或迁移工具（虽有“PostgreSQL 数据库结构更新方案”这类文档，但没有 Flyway/Migration 脚本）。
- 未发现 Dockerfile 或 docker-compose 等部署配置。
- 没有 package.json、requirements.txt、go.mod 等依赖清单。

因此，本仓库**目前处于“数据与文档先行”的阶段**，技术架构部分应理解为未来应用系统的设计蓝图，而非现有事实。

## 1.7 架构演进建议

从当前仓库状态到完整应用系统，可以考虑以下演进路径：

1. 保持“目录+元数据+JSON”作为事实标准
   - 将本仓库视为“单一事实来源”，后续系统开发、数据仓库建模都以此为基准。
2. 引入前后端与数据库工程
   - 在同一仓库或独立仓库中创建前端（React）和后端（Express）项目，将本仓库的数据结构映射为 API/DTO/实体。
3. 将 JSON 数据迁移至数据库
   - 以“员工信息数据表结构”“PostgreSQL 数据库结构更新方案”为基础，编写迁移脚本将 JSON 批量导入 PostgreSQL。
4. 统一分析与报表平台
   - 将 `06_分析数据层级` 中的结果作为数据仓库或分析平台的输出，统一由定时任务或流式处理生成。

本 Code Wiki 的后续章节会从“领域模型与元数据”“业务流程模块”“代码与数据处理流水线”“分析层”等角度进一步细化架构各层级的职责与关系。

## 1.8 三个世界视角

### 1.8.1 原子世界（有空间 + 有时间）

- 架构最终要服务现实组织的运转：招聘、入职、在职管理、离职、培训、薪资等动作都发生在具体的人与具体的时间窗口中。
- 现实世界的约束决定了系统边界：审批链、职责分工、数据采集时点、合规要求会反过来约束信息结构如何设计。

### 1.8.2 信息世界（无空间 + 有时间）

- 本章描述的“元数据层级 / 业务流程层级 / 分析数据层级”本质上都是信息世界的分层组织方式。
- 信息世界的关键问题是可追溯与一致性：目录即事实标准、元数据定义即字段口径，所有业务数据与分析输出都应对齐这些口径。

### 1.8.3 AI 世界（无空间 + 存储层无时间）

- 本章的“架构演进建议”可以理解为从信息世界走向 AI 世界的前置步骤：先固化结构与历史，再引入推理与预测能力。
- AI 能力的落点更适合从分析层扩展：在既有指标与聚合口径之上，逐步加入预测、异常检测与自动化建议。
