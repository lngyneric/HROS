# 07. 术语表与规范约定

## 7.1 关键术语

| 术语             | 含义                                                                 |
|------------------|----------------------------------------------------------------------|
| HR信息生命周期   | 员工从招聘、入职、在职、培训、薪资、绩效、离职到离职后管理的完整过程 |
| 元数据层级       | 定义系统中各类实体（部门、职位、员工信息、流程、表单等）的字段和结构 |
| 分析数据层级     | 基于业务数据生成的统计与分析结果（JSON 报告等）                     |
| 非敏感信息       | 一般业务信息，例如组织架构、职位描述、公开流程配置等                 |
| 敏感信息         | 需特殊保护的信息，如个人身份、薪资、绩效、健康、银行账号等           |
| 在职员工         | 当前在公司任职的员工                                                 |
| 离职员工         | 已完成离职流程的员工                                                 |
| 归档             | 将历史数据（如离职员工信息）封存到不可随意修改的文件或记录中         |
| 归档索引         | 对归档文件进行索引，记录归档时间、数量、文件名等                     |
| 员工主档         | 员工的基础档案信息，包括个人信息、岗位、部门、状态等                 |

## 7.2 目录与文件命名规范

### 7.2.1 目录命名

- 采用“编号_中文名称”的命名方式，例如：
  - `01_元数据层级`
  - `02_招聘流程管理`
  - `03_培训与发展管理`
  - `05_入离职管理`
- 子目录以业务阶段或信息类别命名，例如：
  - `01_职位发布阶段`
  - `02_在职期间管理`
  - `03_离职办理阶段`
  - `非敏感信息` / `敏感信息`

这种命名方式有利于：

- 在文件浏览器中自动按编号排序。
- 仅凭目录名即可大致理解内容。

### 7.2.2 文件命名

- Markdown 文档：
  - 一般使用描述性中文名称，例如“员工信息定义文档.md”“招聘流程管理定义文档.md”。
  - 对于统一入口文档，如“HR信息生命周期管理系统总览.md”，用于承载模块整体说明。
- JSON 数据文件：
  - 对于实体集合类数据，推荐命名为“实体+信息”，如：
    - `在职员工信息.json`
    - `离职员工信息.json`
    - `员工完整信息.json`
    - `员工状态分析报告.json`
  - 对于归档数据，推荐带时间戳，例如：
    - `离职员工归档_2026-02-28T03-36-21.json`

## 7.3 数据字段规范（示例）

结合入离职模块 JSON 与脚本逻辑，以下字段在员工数据中较为关键：

- `雇员编号`
  - 员工唯一标识，必填。
- `姓名`
  - 员工姓名，必填。
- `部门`
  - 员工所在部门，必填；在脚本中会去除空白以标准化。
- `职位`
  - 员工所在职位，在脚本中同样会去除空白。
- `性别`
  - 一般使用“男/女”，脚本中使用该字段统计性别分布。
- `工作地点`
  - 员工工作地点，用于统计地点分布。

脚本中将 `雇员编号/姓名/部门` 视为必填字段，并进行数据质量校验。

## 7.4 敏感信息处理规范（结合系统总览）

根据系统总览文档中“数据安全要求”的说明，在后续系统实现阶段应遵循：

- 访问控制：
  - 使用基于角色的权限控制（RBAC）。
  - 实施最小权限原则并定期审查。
- 数据加密：
  - 使用 HTTPS/TLS 保护传输。
  - 对敏感字段进行存储加密（如 AES-256）。
  - 妥善管理密钥。
- 审计日志：
  - 记录敏感数据的访问和修改操作。
  - 对异常操作进行告警。

在当前“文件+脚本”阶段：

- 需通过文件系统权限、加密磁盘、访问控制等方式保护包含敏感信息的目录。

## 7.5 脚本与分析任务命名建议

对于未来新增的脚本或分析任务，建议：

- 使用功能+动作的英文名称，例如：
  - `employee_data_cleaning.js`
  - `employee_status_analysis.py`
- 在脚本所在目录下增加 README，说明：
  - 输入文件与目录。
  - 输出文件与目录。
  - 主要处理步骤。
  - 与元数据/分析层的关系。

## 7.6 文档维护建议

- 对于重要的“定义文档”和“总览文档”：
  - 建议在文档中维护“版本信息”表，记录版本号、变更日期、变更内容、变更人。
  - 当前的“HR信息生命周期管理系统总览.md”已提供了版本信息表，可作为示例。
- 对于频繁变化的字段或流程：
  - 可先在元数据定义文档中约定变更流程，例如“需要评审后才能调整字段含义”，避免后端/前端/报表各自为政。

通过上述术语与规范约定，本 Code Wiki 希望为后续的系统开发、数据治理和文档扩展提供统一的语言和标准基础。

## 7.7 三个世界视角

### 7.7.1 原子世界（有空间 + 有时间）

- 规范最终要落在人的行为上：谁可以访问敏感信息、谁可以修改字段口径、谁负责审批与复核，这些都是现实组织约束。
- 命名与分级规范的价值在于减少现实协作摩擦：让跨角色沟通时能快速对齐“这是什么、放在哪、谁能看、何时更新”。

### 7.7.2 信息世界（无空间 + 有时间）

- 本章的术语与命名规则是信息世界的“共同语言”：统一目录、文件、字段与归档方式，减少歧义与口径漂移。
- 时间性体现在版本与变更：规范不仅要描述“现在是什么”，也要支持“将来如何变”，例如通过版本信息表与变更记录来实现可追溯。

### 7.7.3 AI 世界（无空间 + 存储层无时间）

- 规范是 AI 的前置条件：术语一致、字段一致、敏感分级清晰，才能形成稳定的特征空间与可治理的数据集。
- 若未来引入 AI 推理与预测，命名规范和版本记录也将用于解释模型输入与输出，帮助定位“模型学到的历史口径”与“当前口径”之间的差异。
