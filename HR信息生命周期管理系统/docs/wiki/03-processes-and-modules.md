# 03. 业务流程模块说明

## 3.1 招聘流程管理模块

目录位置：

- [02_招聘流程管理](file:///Users/cherrych/Documents/trae_projects/HROS/HR信息生命周期管理系统/02_招聘流程管理)

整体结构（与 README/总览文档中的定义一致）：

```text
02_招聘流程管理/
├── 01_职位发布阶段/
│   ├── 招聘需求分析.json
│   ├── 招聘需求分析报告.md
│   └── 职位发布清单.json
└── 招聘流程管理定义文档.md
```

### 3.1.1 模块职责

- 从职位需求提出开始，负责整个招聘链路的数据与文档沉淀：
  - 招聘需求分析：业务部门提出的招聘需求、岗位职责、人数、时间要求等。
  - 职位发布：在不同渠道的职位发布信息与清单。
  - （在定义文档和系统总览中还包含简历筛选、面试评估、录用决策、入职准备等阶段，当前仓库主要落地了“职位发布阶段”的数据文件。）

### 3.1.2 关键文件说明

- 招聘流程管理定义文档.md
  - 描述招聘流程的总体设计、各阶段职责、输入/输出、参与角色等。
- 招聘需求分析.json
  - 结构化记录招聘需求，包括职位名称、所属部门、需求人数、优先级等。
- 招聘需求分析报告.md
  - 对需求分析结果进行汇总与解读，通常面向管理层。
- 职位发布清单.json
  - 记录每个需求在不同招聘渠道上的发布信息（如平台名称、发布时间、预算等）。

### 3.1.3 与元数据/分析层的关系

- 与元数据层级：
  - 职位信息、部门信息、流程节点定义等均来自 01_元数据层级。
- 与分析层：
  - 招聘周期、渠道效果、录用率、招聘成本等分析指标将来源于这些 JSON 数据。

### 3.4.1 模块职责

- 入职办理阶段：
  - 采集入职员工的基础信息、身份信息、职位/部门信息。
  - 通过“员工入职信息定义文档”和“员工信息数据分析报告”来规范和评估数据。
- 在职期间管理：
  - 记录和维护在职员工信息，按部门/职位分组。
  - 提供在职员工统计信息，为分析层和人力决策提供基础数据。
- 离职办理阶段：
  - 管理离职员工的基础信息。
  - 使用脚本进行清洗、归档，并生成“归档索引”和统计报告。
- 横向“非敏感信息”目录：
  - 存放按部门聚合的员工完整信息，用于综合分析。

### 3.4.2 与数据清洗脚本的协同

入离职模块是当前仓库中**唯一包含可执行代码**的模块：

- [data_cleaning_and_archive.js](file:///Users/cherrych/Documents/trae_projects/HROS/HR信息生命周期管理系统/05_入离职管理/data_cleaning_and_archive.js)

脚本的主要目标是：

- 从“在职员工信息.json”和“离职员工信息.json”中读取数据。
- 对关键字段进行清洗和必填校验。
- 将离职员工数据归档到带时间戳的文件中，并维护“归档索引.json”。
- 生成“在职员工信息_清洗后.json”和“在职员工统计信息.json”。
- 输出“数据清洗和归档报告.json”，记录数据质量和输出文件路径。

该脚本的详细说明见本 Wiki 的《04. 代码与数据处理流水线》章节。

## 3.5 模块间依赖关系（概览）

- 元数据层级：
  - 向所有业务流程模块提供“部门/职位/员工字段定义/流程结构”等基础信息。
- 招聘、培训、薪资、入离职：
  - 在业务操作中生成员工/职位/部门相关的事实数据。
  - 这些数据在不同阶段可能需要共享或联动（例如入职信息来自招聘模块的录用结果）。
- 分析数据层级：
  - 以业务流程层级的事实数据为输入，以元数据层级为维度定义，输出各类统计和分析 JSON 报告。

后续如果引入完整的应用系统，应在代码层面明确这些依赖关系，保证：

- 模块间通过清晰的接口或数据库结构解耦。
- 不同模块对同一实体的字段使用一致的元数据定义。

## 3.6 三个世界视角

### 3.6.1 原子世界（有空间 + 有时间）

- 各业务模块本质上描述的是现实世界的动作链：招聘与面试、培训组织与授课、入职办理与交接、离职面谈与结算等。
- “阶段化目录”对应现实世界的顺序依赖：例如先有录用决策，才会进入入职准备；先有离职办理，才会进入离职归档。

### 3.6.2 信息世界（无空间 + 有时间）

- 当前仓库把流程模块落为“可被存档的结构”：阶段目录、定义文档与样例 JSON 是信息世界的主要载体。
- 信息世界的关键工作是补齐链路：让各阶段的数据能按统一字段串起来，形成可追溯的端到端记录，而不是孤立的文件集合。

### 3.6.3 AI 世界（无空间 + 存储层无时间）

- AI 更适合在“流程数据完整之后”介入：基于招聘转化链路做渠道优化，基于入离职链路做离职预测，基于培训记录做效果预测。
- 对尚未落地的数据阶段（如薪资模块）而言，AI 的前置工作是先定义好信息世界的采集与口径，否则难以形成稳定能力。
