## 5.3 分析层与前两层的关系

分析数据层级与其他层级关系如下：

- 元数据层级：
  - 定义部门、职位、员工字段及其含义。
  - 分析报告中的维度（如部门名称、职位名称）应参照元数据中的规范。
- 业务流程层级：
  - 提供事实数据，例如在职/离职员工记录、新员工入职记录等。
  - 分析任务从这些 JSON 或未来数据库表中抽取数据进行聚合。

当系统演进到服务化/数据库化阶段：

- 分析层可以迁移至数据仓库或专门的分析数据库。
- 现有 JSON 报告可以视为报表导出或接口返回的示例。

## 5.4 分析维度与指标示例

结合系统总览中对分析模块的规划，可以对“分析维度”和“指标”做一个通用说明。

### 5.4.1 招聘分析

- 维度：
  - 部门、职位、招聘渠道、时间。
- 指标：
  - 招聘周期、录用率、渠道转化率、招聘成本、offer 接受率等。

### 5.4.2 培训效果分析

- 维度：
  - 培训项目、部门、职位、讲师、时间。
- 指标：
  - 培训覆盖率、满意度、考试通过率、知识掌握度、培训 ROI 等。

### 5.4.3 薪资福利分析（规划）

- 维度：
  - 部门、职位、职级、地区、性别等。
- 指标：
  - 薪资结构比例、福利成本占比、薪资竞争力指数、薪资公平性指标等。

### 5.4.4 绩效评估分析（规划）

- 维度：
  - 部门、职位、考核周期、绩效等级。
- 指标：
  - 绩效分布、绩效趋势、部门绩效对比、绩效与离职的关联等。

### 5.4.5 离职分析

- 维度：
  - 部门、职位、年龄、工龄、离职原因、时间。
- 指标：
  - 离职率、离职原因分布、关键人群流失率、离职成本等。

当前仓库中的 JSON 分析文件可以看作这些分析的部分实现或样例结果。

## 5.5 分析层落地建议

在后续演化中，可以考虑：

- 将“在职员工统计信息.json”等统计结果迁移到分析数据层级，保持目录语义统一。
- 使用定时任务（如 cron 或调度平台）定期运行清洗与统计脚本，将结果写入统一分析目录或数据库。
- 在前端系统中构建对应的数据可视化页面，用图表展示这些 JSON 中的指标。

## 5.6 三个世界视角

### 5.6.1 原子世界（有空间 + 有时间）

- 分析需求来源于现实决策场景：什么时候要看人员结构、什么时候要看离职趋势、哪些指标用于绩效复盘或管理评审。
- 指标的解释依赖现实语境：同一个“离职率”在不同时间窗口、不同组织边界下含义不同，需要与业务动作对齐。

### 5.6.2 信息世界（无空间 + 有时间）

- 分析层的核心产物是信息世界的“可读结论”：用统一口径把事实数据聚合成报告与指标 JSON。
- 时间性体现在报告的生成频率与口径一致性：报告何时更新、使用哪一版数据源、是否可追溯到归档文件与清洗输出。

### 5.6.3 AI 世界（无空间 + 存储层无时间）

- 分析层是 AI 最自然的起点：已有指标与维度可作为特征空间设计的参照，报告结构可作为模型输出的目标形态。
- 从“统计”走向“预测”的路径通常是：先固化口径与数据链路，再引入预测、异常检测和自动化建议，最后形成可回写的信息世界产物。
