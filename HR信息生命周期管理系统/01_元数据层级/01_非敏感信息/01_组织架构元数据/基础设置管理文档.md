## 一、基础设置概述

### 1.1 基础设置定义
基础设置是组织架构管理系统的底层配置模块，用于定义组织架构的核心属性、规则和标准，为其他模块提供支撑。

### 1.2 基础设置的重要性
1. **标准化管理**: 建立统一的组织架构标准和规范
2. **系统支撑**: 为组织架构、职位架构等模块提供基础数据
3. **数据一致性**: 确保组织架构数据的一致性和准确性
4. **灵活性**: 通过配置而非编码实现业务规则的调整

### 1.3 基础设置模块
- 组织架构层级主档
- 组织架构层级设定
- 组织架构属性
- 职位名称主档

## 二、组织架构层级主档

### 2.1 功能说明
定义组织架构的层级结构，是组织架构的核心框架。

### 2.2 层级定义

#### 标准层级结构
```
Level 1: 集团总部（Group/Headquarters）
Level 2: 事业部/分公司（Business Unit/Branch）
Level 3: 部门（Department）
Level 4: 科室/班组（Section/Team）
Level 5: 小组（Group）
```

#### 层级详细说明

##### Level 1: 集团总部
- **定义**: 企业最高管理层和核心职能部门
- **示例**: 总经办、战略部、财务部、人力资源部
- **管理范围**: 全公司
- **决策权限**: 战略决策、重大投资、高层任免

##### Level 2: 事业部/分公司
- **定义**: 独立运营的业务单元或地理区域
- **示例**: 华东事业部、华南分公司、技术事业部
- **管理范围**: 本事业部/分公司
- **决策权限**: 业务决策、人员招聘、预算管理

##### Level 3: 部门
- **定义**: 承担特定职能或业务的工作单元
- **示例**: 销售部、市场部、研发部、客服部
- **管理范围**: 本部门
- **决策权限**: 部门管理、任务分配、绩效评估

##### Level 4: 科室/班组
- **定义**: 部门内的专业小组或工作班组
- **示例**: 技术支持组、客户服务组、质量检验班
- **管理范围**: 本科室/班组
- **决策权限**: 日常工作安排、技术指导

##### Level 5: 小组
- **定义**: 最小的工作单元
- **示例**: 项目小组、临时工作组
- **管理范围**: 本小组
- **决策权限**: 任务执行、问题解决

### 2.3 数据结构

#### 层级主档表
| 字段名 | 字段类型 | 字段长度 | 是否必填 | 说明 |
|--------|----------|----------|----------|------|
| level_id | VARCHAR | 50 | 是 | 层级唯一标识 |
| level_name | VARCHAR | 100 | 是 | 层级名称 |
| level_code | VARCHAR | 20 | 是 | 层级编码 |
| level_order | INT | - | 是 | 层级顺序 |
| level_description | VARCHAR | 500 | 否 | 层级描述 |
| parent_level_id | VARCHAR | 50 | 否 | 上级层级ID |
| max_departments | INT | - | 否 | 最大部门数量 |
| create_date | DATETIME | - | 是 | 创建日期 |
| status | CHAR | 1 | 是 | 状态（A-有效/I-无效） |

### 2.4 层级配置示例
```json
{
  "levels": [
    {
      "level_id": "L1",
      "level_name": "集团总部",
      "level_code": "HQ",
      "level_order": 1,
      "max_departments": 20
    },
    {
      "level_id": "L2",
      "level_name": "事业部",
      "level_code": "BU",
      "level_order": 2,
      "max_departments": 10
    },
    {
      "level_id": "L3",
      "level_name": "部门",
      "level_code": "DEPT",
      "level_order": 3,
      "max_departments": 50
    }
  ]
}
```

## 三、组织架构层级设定

### 3.1 功能说明
设置组织架构层级的具体规则，用于规范组织架构的构建和管理。

### 3.2 配置项说明

#### 3.2.1 层级数量限制
- **最大层级数**: 5级（可根据企业规模调整）
- **最小层级数**: 2级（至少包含总部和部门）
- **推荐层级数**: 3-4级（适合大多数企业）

#### 3.2.2 层级命名规则
| 层级 | 命名规则 | 示例 |
|------|----------|------|
| Level 1 | XX总部/总经办 | 集团总部、总经办 |
| Level 2 | XX事业部/XX分公司 | 华东事业部、华南分公司 |
| Level 3 | XX部 | 销售部、市场部、研发部 |
| Level 4 | XX科/XX组 | 技术科、客服组 |
| Level 5 | XX小组 | 项目小组 |

#### 3.2.3 层级关系规则
- **父子关系**: 每个层级只能有唯一的上级层级
- **跨级限制**: 不允许跨级创建部门（如Level 1直接创建Level 3部门）
- **同级互斥**: 同一层级的部门之间是平级关系

#### 3.2.4 层级权限规则
| 层级 | 创建权限 | 修改权限 | 删除权限 | 查看权限 |
|------|----------|----------|----------|----------|
| Level 1 | 系统管理员 | 系统管理员 | 系统管理员 | 全员 |
| Level 2 | 系统管理员 | 高层管理 | 系统管理员 | 全员 |
| Level 3 | 高层管理 | 中层管理 | 高层管理 | 全员 |
| Level 4 | 中层管理 | 基层管理 | 中层管理 | 全员 |
| Level 5 | 基层管理 | 基层管理 | 基层管理 | 全员 |

### 3.3 数据结构

#### 层级设定表
| 字段名 | 字段类型 | 字段长度 | 是否必填 | 说明 |
|--------|----------|----------|----------|------|
| setting_id | VARCHAR | 50 | 是 | 设定唯一标识 |
| setting_name | VARCHAR | 100 | 是 | 设定名称 |
| setting_code | VARCHAR | 50 | 是 | 设定编码 |
| setting_value | VARCHAR | 500 | 是 | 设定值 |
| value_type | VARCHAR | 20 | 是 | 值类型（NUMBER/STRING/JSON/BOOLEAN） |
| setting_description | VARCHAR | 500 | 否 | 设定描述 |
| is_system | CHAR | 1 | 是 | 是否系统设定（Y/N） |
| create_date | DATETIME | - | 是 | 创建日期 |
| update_date | DATETIME | - | 是 | 更新日期 |

### 3.4 配置示例
```json
{
  "settings": [
    {
      "setting_code": "MAX_LEVEL",
      "setting_name": "最大层级数",
      "setting_value": "5",
      "value_type": "NUMBER"
    },
    {
      "setting_code": "NAMING_RULE",
      "setting_name": "层级命名规则",
      "setting_value": "{\"L1\":\"XX总部\",\"L2\":\"XX事业部\",\"L3\":\"XX部\"}",
      "value_type": "JSON"
    },
    {
      "setting_code": "ALLOW_SKIP_LEVEL",
      "setting_name": "允许跨级创建",
      "setting_value": "false",
      "value_type": "BOOLEAN"
    }
  ]
}
```

## 四、组织架构属性

### 4.1 功能说明
配置组织架构的附加属性，丰富组织架构的信息维度。

### 4.2 标准属性

#### 4.2.1 基本属性
| 属性名称 | 属性类型 | 是否必填 | 说明 |
|----------|----------|----------|------|
| 部门类型 | 枚举 | 是 | 成本中心/利润中心/支持部门 |
| 部门职责 | 文本 | 是 | 部门主要职责描述 |
| 成立日期 | 日期 | 是 | 部门成立日期 |
| 部门人数 | 数字 | 否 | 部门总人数 |
| 办公地点 | 文本 | 否 | 部门办公地点 |

#### 4.2.2 管理属性
| 属性名称 | 属性类型 | 是否必填 | 说明 |
|----------|----------|----------|------|
| 预算权限 | 布尔 | 否 | 是否有预算审批权限 |
| 招聘权限 | 布尔 | 否 | 是否有独立招聘权限 |
| 培训权限 | 布尔 | 否 | 是否有培训组织权限 |
| 考核权限 | 布尔 | 否 | 是否有绩效考核权限 |

#### 4.2.3 业务属性
| 属性名称 | 属性类型 | 是否必填 | 说明 |
|----------|----------|----------|------|
| 业务范围 | 文本 | 否 | 部门负责的业务范围 |
| 服务对象 | 文本 | 否 | 部门服务的客户群体 |
| 关键指标 | 文本 | 否 | 部门的关键绩效指标 |
| 协作部门 | 多选 | 否 | 常协作的其他部门 |

### 4.3 自定义属性
企业可根据实际需要添加自定义属性：

#### 示例：销售部门特殊属性
```json
{
  "custom_attributes": [
    {
      "attribute_name": "销售区域",
      "attribute_code": "sales_region",
      "attribute_type": "ENUM",
      "options": ["华东", "华南", "华北", "华中", "西部"]
    },
    {
      "attribute_name": "销售目标",
      "attribute_code": "sales_target",
      "attribute_type": "NUMBER",
      "unit": "万元"
    },
    {
      "attribute_name": "客户数量",
      "attribute_code": "customer_count",
      "attribute_type": "NUMBER"
    }
  ]
}
```

### 4.4 数据结构

#### 属性定义表
| 字段名 | 字段类型 | 字段长度 | 是否必填 | 说明 |

| attribute_name | VARCHAR | 100 | 是 | 属性名称 |

| attribute_type | VARCHAR | 20 | 是 | 属性类型 |

| sort_order | INT | - | 是 | 排序顺序 |
| create_date | DATETIME | - | 是 | 创建日期 |

|----------|----------|----------|----------|

| Receptionist | 前台文员 | P4 | 接待工作 |
| Secretary | 秘书 | P3 | 文秘工作 |

### 5.3 职位名称规范

#### 5.3.1 命名规则
1. **管理类**: 职能+级别（如：销售经理、技术总监）
2. **技术类**: 级别+职能（如：高级工程师、技术专家）
3. **专业类**: 职能+级别（如：财务专员、法务经理）
4. **行政类**: 职能+级别（如：行政助理、前台文员）

#### 5.3.2 禁用词汇
- 简称（如：经理→部门经理）
- 俗称（如：老板→总经理）
- 英文缩写（如：PM→项目经理）
- 方言词汇

## 六、配置管理

### 6.1 配置项管理
1. **配置项分类**: 系统配置、业务配置、用户配置
2. **配置项维护**: 新增、修改、删除、查询
3. **配置项导入导出**: 支持批量导入导出
4. **配置项版本管理**: 保留历史版本

### 6.2 配置生效规则
1. **立即生效**: 系统配置类，修改后立即生效
2. **定时生效**: 业务配置类，可设置生效时间
3. **审批生效**: 重要配置，需要审批后生效
4. **灰度发布**: 大范围变更，支持灰度发布

### 6.3 配置回滚
1. **版本回滚**: 回滚到指定历史版本
2. **配置备份**: 定期自动备份配置
3. **变更记录**: 记录所有配置变更历史

## 七、系统接口

### 7.1 配置查询接口
- **获取层级列表**: `GET /api/settings/levels`
- **获取层级设定**: `GET /api/settings/level-settings`
- **获取属性列表**: `GET /api/settings/attributes`
- **获取职位名称**: `GET /api/settings/job-titles`

### 7.2 配置维护接口
- **新增配置项**: `POST /api/settings`
- **修改配置项**: `PUT /api/settings/{id}`
- **删除配置项**: `DELETE /api/settings/{id}`
- **批量导入**: `POST /api/settings/import`

### 7.3 配置导出接口
- **导出Excel**: `GET /api/settings/export/excel`
- **导出JSON**: `GET /api/settings/export/json`
- **导出XML**: `GET /api/settings/export/xml`

## 八、注意事项

### 8.1 配置管理
1. **谨慎修改**: 系统配置修改可能影响全局
2. **测试验证**: 重大变更先在测试环境验证
3. **备份恢复**: 修改前做好备份，出现问题及时恢复
4. **文档更新**: 配置变更后及时更新文档

### 8.2 权限控制
1. **分级授权**: 不同级别配置由不同权限人员管理
2. **审批流程**: 重要配置变更需要审批
3. **操作日志**: 记录所有配置操作日志
4. **审计跟踪**: 定期审计配置变更记录

### 8.3 数据质量
1. **数据校验**: 确保配置数据的准确性
2. **一致性检查**: 检查配置之间的一致性
3. **定期清理**: 清理无效的配置数据
4. **版本管理**: 保持配置的版本一致性
